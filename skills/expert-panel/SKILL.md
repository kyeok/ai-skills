# /전문가소환 - 근거 기반 전문가 패널 `v2.0`

$ARGUMENTS

주어진 주제에 대해 **관련 학문 분야의 근거를 먼저 탐색**한 후, 해당 근거에 기반한 전문가 패널 토론을 진행합니다.

> 설계 원칙: "전문가가 이렇다고 해서 무조건 따르지 않는다. 근거 없는 전문가 조언은 안 하느니만 못할 수 있다." (EBP - Evidence-Based Practice)
>
> 영감: 김창준의 EBP(Evidence-Based Practice) 접근법에서 영향받았습니다.

## 사용법

```
/전문가소환 [주제]                                    → 전체 프로세스 실행
/전문가소환 [주제] --experts [전문가1, 전문가2, ...]   → 전문가 직접 지정
/전문가소환 [주제] --quick                             → 근거 탐색 생략, 빠른 토론
```

---

## 실행 단계

### 1단계: 학문 지도 그리기 (Science Mapping)

`$ARGUMENTS`에서 주제를 파악한 후, **관련 학문 분야를 먼저 식별**한다.

수행 항목:
- 주제와 관련된 학문 분야(XX Science, XX Psychology, XX Studies 등) **2-4개** 식별
- 각 분야에서 이 주제에 대해 어떤 종류의 연구가 축적되어 있는지 개요 제시
- 가장 관련성 높은 1-2개 분야를 **주 근거원(primary evidence source)**으로 선택

출력 형식:
```markdown
## 학문 지도

| 분야 | 관련성 | 핵심 연구 주제 | 근거 풍부도 |
|-----|--------|---------------|-----------|
| [분야명] | 높음/중간 | [이 분야에서 다루는 관련 주제] | 풍부/보통/희소 |

**주 근거원**: [선택한 1-2개 분야와 선택 이유]
```

### 2단계: 근거 지형 탐색 (Evidence Landscape)

주 근거원 분야에서 **핵심 발견들을 정리**한다.

수행 항목:
- **현상 이해**: 이 현상이 어떻게 작동하는가? 무엇이 관찰되는가?
- **실천 방법**: 어떻게 하면 더 효과적인가? 무엇이 검증되었는가?
- 각 발견에 **근거 수준** 태그 부여

근거 수준 체계:
- **[Strong]** 메타분석, 체계적 리뷰, 복수의 재현된 실험
- **[Moderate]** 복수의 RCT 또는 대규모 관찰 연구
- **[Emerging]** 단일 연구, 예비 결과, 파일럿 스터디
- **[Expert]** 전문가 합의, 임상 경험 기반 (근거 약함 - 주의 필요)
- **[Contested]** 상반된 연구 결과가 존재

출력 형식:
```markdown
## 근거 지형

### 현상 이해 (이렇게 작동한다)
- [Strong] 발견 내용 — 근거 출처
- [Moderate] 발견 내용 — 근거 출처

### 실천 방법 (이렇게 하면 효과적이다)
- [Strong] 발견 내용 — 근거 출처
- [Emerging] 발견 내용 — 근거 출처

### 통념 경고 (근거 없이 퍼진 믿음)
- [Contested] 흔히 믿지만 근거가 약하거나 반증된 것들
```

> **통념 경고란?** 전문가 권위만으로 퍼졌지만 실제 근거가 약하거나 역효과가 검증된 통념을 명시적으로 플래그한다. 근거 없는 "상식"이 가장 위험하다.

### 3단계: 컨텍스트 수집

사용자의 **구체적 상황과 맥락**을 파악한다.

수행 항목:
- 이전 대화 맥락에서 관련 정보 추출
- NDM(Naturalistic Decision Making)의 RPD(Recognition-Primed Decision) 모델로 상황 인식 강화
- 필요시 사용자에게 **2-3개 핵심 질문** (현재 상황, 제약 조건, 원하는 변화)

이 단계의 핵심: 4단계 전문가 토론이 **사용자의 실제 맥락에 맞춰** 진행되도록 정보 수집.

### 4단계: 근거 기반 패널 토론

**근거 지형 위에서** 전문가 패널을 소환하여 토론한다.

#### 전문가 선정 (6인)

선정 기준 (**권위가 아닌 근거 기여도** 기준):
- 해당 분야에서 핵심 연구를 수행한 연구자 (최소 3인)
- 해당 근거를 실무에 적용한 실천가 (최소 1인)
- 다른 분야에서 교차 관점을 제공할 수 있는 전문가 (최소 1인)
- **Implementation 전문가** 또는 **비판적 관점** 보유자 (최소 1인)

`--experts` 플래그로 사용자 직접 지정 가능 (이 경우에도 근거 기반 토론 원칙은 유지).

#### 토론 구조

1. **전문가 소개**: 이름, 전문 분야, **핵심 연구 기여** (저서보다 연구 기여 우선)
2. **근거 기반 라운드**: 2단계에서 발견한 근거에 대해 각 전문가가 해석/확장/반박
3. **사용자 맥락 적용 라운드**: 이 근거가 사용자의 구체적 상황에서 어떻게 적용되는가
4. **교차 검증**: 전문가 간 의견 충돌 시 **어떤 근거에 기반한 의견인지** 명시하며 토론

#### 토론 규칙
- 모든 주장에 근거 수준 태그 필수: `[Strong]`, `[Moderate]`, `[Emerging]`, `[Expert]`, `[Contested]`
- `[Expert]` 수준의 의견은 반드시 "이것은 근거가 제한적이며, 제 경험에 기반한 의견입니다"를 명시
- 만장일치더라도 근거가 약하면 그 사실을 솔직하게 표기

### 5단계: 구현 설계 (Implementation Design)

Implementation Science 원칙을 적용하여 **실제로 작동하는 실행 계획**을 만든다.

> "좋은 베스트프랙티스도 구현 방법이 잘못되면 실패한다" — 빌 게이츠 재단 탁월한 교사 프로젝트 실패 사례

수행 항목:

```markdown
## 구현 설계

### Core Practice (절대 바꾸면 안 되는 핵심)
- 이 원칙/방법의 효과를 만드는 최소한의 핵심 요소들
- 이것을 빼거나 바꾸면 효과가 사라짐

### Peripheral Practice (맥락에 맞게 변형 가능)
- 핵심 기능은 유지하되 형태는 자유롭게 적응 가능한 요소들
- 사용자의 환경/제약에 맞춘 변형 제안

### 적응 설계 (Adaptation)
- 사용자의 구체적 맥락(직업, 환경, 도구, 습관)에 맞춘 구현 방법
- "원본 그대로 이식"이 아닌 "원리를 보존한 적응"

### 이해관계자 고려
- 이 변화가 영향을 미치는 사람들
- 그들의 참여/동의가 필요한 부분

### 퍼실리테이션 포인트
- 혼자 하기 어려운 부분, 도움이 필요한 부분
- 활용 가능한 도구/자원 (앱, 커뮤니티, 코칭 등)
```

### 6단계: 실천 실험 설계 (PBE Loop)

EBP(근거→실천)에서 PBE(실천→근거)로 이어지는 **선순환 구조**를 설계한다.

```markdown
## 실천 실험

### 최소 실험 (Minimum Viable Experiment)
- 1-2주 안에 시도할 수 있는 가장 작은 실험
- 명확한 성공/실패 판단 기준
- 측정 방법 (주관적 체감 + 가능하면 객관적 지표)

### 관찰 포인트
- 실험하면서 주의 깊게 관찰할 것들
- 예상과 다른 결과가 나올 수 있는 지점

### 반복 주기
- 언제 실험을 평가하고 조정할 것인가
- 효과 있으면 → 확장 방법
- 효과 없으면 → 대안 탐색 또는 적응 재설계
```

### 7단계: Obsidian 저장

모든 단계가 완료되면, 결과물 전체를 **Obsidian vault에 마크다운 파일로 저장**한다.

저장 경로: `$STUDYVAULT_PATH/전문가소환/` (기본값: `~/StudyVault`)

파일명 규칙: `YYYY-MM-DD [주제 요약].md`

저장 내용:
- 1~6단계의 전체 결과물을 하나의 마크다운 파일로 통합
- 파일 상단에 Obsidian frontmatter(YAML) 추가:

```yaml
---
title: "[주제 요약]"
date: YYYY-MM-DD
type: 전문가소환
tags:
  - 전문가소환
  - [관련 태그 1-3개]
experts: [소환된 전문가 이름 목록]
evidence_level: [사용된 근거 수준 중 가장 높은 것]
---
```

- 저장 후 사용자에게 파일 경로를 알려준다
- `전문가소환/` 디렉토리가 없으면 자동 생성한다

## --quick 모드

`--quick` 플래그 사용 시 1-2단계(학문 지도 + 근거 지형)를 생략하고 바로 전문가 토론으로 진입.
단, 토론 중 근거 수준 태그는 여전히 사용하며, `[Expert]` 수준 의견에 대한 경고도 유지.

---

## 톤 & 스타일

- 각 전문가의 실제 어투와 사상을 최대한 반영
- 전문 용어는 영어 OK, 설명은 한국어
- **비판단적이고 호기심 어린 톤** — 근거가 약한 것을 "틀렸다"가 아니라 "아직 검증이 부족하다"로 표현
- 실용적: 토론으로 끝나지 않고 반드시 **구현 가능한 실험**으로 연결
- 정직: 근거가 없으면 없다고 솔직하게 말한다
